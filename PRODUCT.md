# Oh My Notion 产品文档

## 1. 产品概述

`Oh My Notion` 是一个使用 Python 实现的 `local-first` 知识助手，目标是优先查询本地同步的 Notion 内容，再基于检索结果生成可信回答，帮助用户快速查找、理解和复用个人知识库。

这个产品不是一个泛化聊天机器人，而是一个面向个人知识管理的 Agent。它的核心能力不是“自由对话”，而是：

- 优先从本地 Notion 知识中检索答案
- 基于检索证据生成回答，而不是直接猜测
- 在本地结果不足时，按规则决定是否回退到远程 Notion
- 用尽量简单的工程结构帮助开发者学习 Agent 的核心实现方式

## 2. 产品目标

### 2.1 用户目标

用户希望获得一个工具，能够：

- 快速搜索自己的 Notion 知识，而不是手动打开多个页面查找
- 用自然语言提问，例如“我之前写过哪些关于 Agent 路由的笔记？”
- 获得基于 Notion 内容整理后的答案，而不是一堆原始搜索结果
- 看到答案来自哪些 Notion 页面，便于验证和继续阅读

### 2.2 开发目标

这个项目还承担一个学习目标：帮助开发者快速理解 Agent 的最小闭环。项目应尽量覆盖以下能力：

- 数据同步
- 文本解析
- 本地全文检索
- 工具调用
- 路由决策
- 基于证据的回答生成
- 可观测与可调试

## 3. 为什么做这个产品

Notion 本身适合记录知识，但不适合在大量内容里做“快速、上下文化、可归纳”的问答。用户经常面临这些问题：

- 知道自己写过，但记不清具体在哪一页
- 搜索只返回关键词匹配，缺少整合后的答案
- 笔记分散在页面、子页面、数据库记录中，人工浏览成本高

`Oh My Notion` 解决的问题是：

- 把 Notion 从“内容存储系统”升级成“本地优先的可问答知识系统”
- 让用户通过一个 Agent 访问自己的知识库
- 用一个足够小的 Python 项目，作为 Agent 开发的学习样板

## 4. 目标用户

主要面向两类用户：

- 个人知识管理重度用户：长期使用 Notion 记录笔记、项目、技术资料
- 想快速学习 Agent 开发的开发者：希望从一个具体、真实且可控的项目入手

## 5. 核心原则

### 5.1 Local-First

查询时优先检索本地索引，不把远程 Notion API 作为默认路径。这样做有几个原因：

- 本地检索更快
- 成本更低
- 可离线工作
- 更容易调试
- 更适合做 Agent 的可控实验

### 5.2 Evidence-Based

Agent 必须基于检索出的页面或片段回答问题。没有足够证据时，应明确说明不知道或证据不足，避免幻觉。

### 5.3 Simple First

第一版不追求复杂的多 Agent 系统，不追求全自动，也不追求花哨 UI。优先做出一个结构清晰、便于理解的最小系统。

### 5.4 Learnable by Building

每个模块都应该足够小、边界清楚，让开发者可以通过阅读和修改代码快速理解 Agent 的组成。

## 6. 产品愿景

第一阶段，它是一个本地优先的 Notion 问答工具。

后续可以逐步演进为：

- 个人知识助手
- 笔记探索和复盘助手
- 写作辅助上下文源
- 个人知识工作流入口

## 7. 使用场景

### 7.1 场景一：查找过去写过的内容

用户提问：

`我之前写过哪些和 agent routing 有关的笔记？`

系统行为：

- 在本地索引中检索相关页面和片段
- 汇总相关笔记主题
- 返回页面标题、摘要和来源链接

### 7.2 场景二：基于已有知识回答问题

用户提问：

`我之前关于 RAG 的笔记里，是怎么区分 retrieval 和 rerank 的？`

系统行为：

- 检索相关页面
- 定位关键段落
- 基于检索证据生成回答
- 附带引用页面

### 7.3 场景三：找不到答案时谨慎处理

用户提问：

`我去年写过 MCP server 的部署流程吗？`

系统行为：

- 优先查本地
- 若命中不足，则说明本地知识库中未找到足够证据
- 可选触发远程同步后重试

## 8. 功能范围

### 8.1 MVP 包含

- 从 Notion API 同步页面到本地
- 将页面内容标准化并存储到本地
- 将文本切分并建立 SQLite 全文索引
- 提供一个本地问答入口
- 回答前执行本地检索
- 基于检索结果生成回答
- 返回引用页面标题和链接
- 记录查询链路，便于调试

### 8.2 MVP 不包含

- 多用户系统
- 复杂权限管理
- 在线协作
- 浏览器插件
- 自动写回 Notion
- 多 Agent 编排
- 复杂向量数据库集群

## 9. 产品流程

### 9.1 离线数据流程

1. 用户配置 Notion 集成 Token 和目标页面/数据库
2. 系统调用 Notion API 拉取页面和块数据
3. 原始响应保存到本地 `data/raw/`
4. 解析为统一的页面文本结构
5. 对文本进行切分，写入 SQLite
6. 建立 FTS 索引

### 9.2 在线查询流程

1. 用户输入问题
2. Agent 调用本地搜索工具
3. 获取 top-k 页面片段
4. 判断证据是否充足
5. 若充足，则生成回答并附引用
6. 若不足，则返回“未找到足够证据”或触发同步策略

## 10. 功能模块设计

### 10.1 Sync 模块

职责：

- 从 Notion API 拉取页面、块、数据库项
- 跟踪更新时间
- 增量同步到本地

输入：

- Notion token
- 起始页面/数据库 ID

输出：

- 本地原始 JSON 文件

### 10.2 Parse 模块

职责：

- 将 Notion block 树转换成统一文本
- 提取标题、正文、标签、时间、URL

输出：

- 规范化页面对象

### 10.3 Index 模块

职责：

- 将页面切分为 chunk
- 存储到 SQLite
- 使用 FTS5 建全文检索索引

输出：

- 可供本地搜索的结构化数据

### 10.4 Tools 模块

职责：

- 为 Agent 提供可调用工具

建议提供的工具：

- `search_local_notion(query, top_k)`
- `get_page(page_id)`
- `list_recent_pages(limit)`
- `sync_notion()`

### 10.5 Agent 模块

职责：

- 解释用户请求
- 调用本地检索工具
- 判断检索结果是否足够回答问题
- 将证据组织成最终答案

### 10.6 Interface 模块

MVP 建议先提供：

- CLI

后续可扩展：

- FastAPI HTTP 接口
- Web UI

## 11. 技术方案

## 11.1 技术选型

首选 Python，原因：

- AI/Agent 生态成熟
- 更适合做同步、清洗、索引、实验
- 学习资料多，原型迭代快

建议依赖：

- `Python 3.11+`
- `uv` 或 `pip` 管理依赖
- `notion-client` 访问 Notion API
- `sqlite3` 存储数据
- `SQLite FTS5` 做全文检索
- `pydantic` 定义数据模型
- `typer` 或 `argparse` 做 CLI
- `FastAPI` 作为后续接口层
- OpenAI 兼容 SDK 用于回答生成

### 11.2 为什么第一版不用向量数据库

因为这个项目的首要目标是学习 Agent，而不是堆叠 AI 基础设施。MVP 先用 SQLite FTS5，原因如下：

- 足够快
- 部署简单
- 易理解
- 易调试
- 有利于先学明白“检索 -> 证据 -> 回答”的闭环

向量检索可以作为第二阶段升级项。

## 12. 数据设计

### 12.1 页面表 `pages`

建议字段：

- `id`
- `title`
- `url`
- `source_type`
- `created_time`
- `last_edited_time`
- `raw_json_path`

### 12.2 片段表 `chunks`

建议字段：

- `chunk_id`
- `page_id`
- `heading`
- `content`
- `position`
- `token_count`

### 12.3 检索索引

对 `chunks.content` 建立 FTS5 索引，支持：

- 关键词查询
- 标题加权
- 简单排序

## 13. Agent 行为设计

### 13.1 系统角色

Agent 是一个“本地知识检索助手”，不是通用百科。它必须遵守以下规则：

- 优先使用本地 Notion 检索
- 不基于空白记忆回答
- 回答必须尽量引用来源
- 当证据不足时，明确表达不确定

### 13.2 决策逻辑

建议第一版使用简单规则，而不是复杂 planner：

1. 收到问题后先检索本地
2. 如果相关结果足够，则回答
3. 如果结果不足，则提示未找到足够证据
4. 可选触发远程同步后再次检索

### 13.3 输出格式

回答建议包含：

- 直接结论
- 基于哪些页面得出结论
- 相关页面链接

## 14. 非功能要求

### 14.1 性能

- 本地搜索响应尽量控制在 1 秒内
- 小型个人知识库应可在普通笔记本上运行

### 14.2 可调试性

系统需要记录：

- 用户问题
- 检索关键词
- 命中的 chunk
- 最终引用页面

### 14.3 可扩展性

后续应支持：

- 混合检索
- rerank
- 多数据源接入
- Web UI

## 15. 里程碑规划

### M1: 本地数据闭环

目标：

- 可以从 Notion 拉取内容到本地
- 可以解析页面正文

交付：

- `sync` 模块
- 原始 JSON 存档
- 统一页面模型

### M2: 本地检索闭环

目标：

- 可以用自然语言查到相关笔记

交付：

- SQLite 表结构
- chunk 切分
- FTS5 搜索

### M3: Agent 问答闭环

目标：

- 可以基于检索结果生成可靠回答

交付：

- `tools` 模块
- `agent` 模块
- CLI `ask`

### M4: 可用性增强

目标：

- 更像一个可持续使用的工具

交付：

- 增量同步
- 最近页面查询
- 简单 HTTP API

## 16. 成功标准

这个产品在 MVP 阶段的成功，不以“智能程度”衡量，而以以下标准衡量：

- 用户能用自然语言查到自己的 Notion 内容
- 回答大多基于本地证据，而不是模型猜测
- 查询结果能附带来源，方便验证
- 开发者可以通过这个项目清楚理解 Agent 的核心组成

## 17. 风险与限制

### 17.1 Notion API 限制

- 权限受集成授权范围影响
- 页面结构复杂，block 解析需要持续补齐

### 17.2 本地索引质量

- 第一版只用全文检索时，语义召回能力有限
- chunk 切分策略会显著影响回答质量

### 17.3 幻觉风险

- 如果没有强制“基于证据作答”，模型可能编造内容

解决思路：

- 严格的系统提示
- 明确的引用要求
- 无证据时拒答

## 18. 后续演进方向

MVP 完成后，可以继续扩展：

- 向量检索
- 混合检索和 rerank
- 页面摘要缓存
- 写作辅助
- 与本地文件系统、Git 仓库联合检索
- Web UI 和聊天界面

## 19. 项目定位总结

`Oh My Notion` 的定位不是“又一个聊天机器人”，而是：

一个使用 Python 实现的、本地优先的、面向个人知识库的 Notion 检索增强 Agent。

它的产品价值在于：

- 帮助用户更快利用自己的知识资产
- 帮助开发者用一个小而完整的项目掌握 Agent 开发的核心方法
